= How Neo4j + Databricks Work Together

Databricks and Neo4j are complementary platforms connected through governed data pipelines. Databricks governs and refines data at scale and answers aggregation questions through SQL. Neo4j makes the connections between entities explicit and answers traversal questions through Cypher. Delta Lake serves as the interchange format between them, the Spark Connector moves data in both directions, and Mosaic AI powers the agents that route questions to the right engine.

This page covers the architecture that joins the two platforms: the Lakehouse foundation, the Spark Connector bridge, and the five integration patterns that carry data and queries between the systems. For the dataset this architecture operates on and the lab roadmap, see the xref:workshop-overview.adoc[Workshop Overview]. For hands-on implementation, see xref:lab2.adoc[Lab 2] (Spark Connector ETL), xref:lab3.adoc[Lab 3] (semantic search), and xref:lab4.adoc[Lab 4] (agents).

== Two Platforms, Two Query Shapes

Databricks answers "how much?" and "how often?" through SQL aggregation over rows and columns. Neo4j answers "how is this connected?" and "what is affected?" through Cypher traversal over nodes and relationships. Most real-world questions require both.

An aircraft maintenance investigation illustrates why. A fleet manager asks: "Which aircraft models have the highest EGT volatility, and what maintenance events are connected to those aircraft?" The first half is a `GROUP BY` with `STDDEV` over the `sensor_readings` table. The second half is a multi-hop traversal from Aircraft through System, Component, and MaintenanceEvent nodes. Neither platform can answer the full question alone. A supervisor agent decomposes the question, routes each half to the platform built for it, and synthesizes a single response.

.Databricks SQL: the aggregation half
[source,sql]
----
SELECT a.model, sen.type,
       AVG(r.value) AS avg_reading,
       STDDEV(r.value) AS volatility
FROM sensor_readings r
  JOIN sensors sen ON r.sensor_id = sen.sensor_id
  JOIN systems s ON sen.system_id = s.system_id
  JOIN aircraft a ON s.aircraft_id = a.aircraft_id
WHERE sen.type = 'EGT'
GROUP BY a.model, sen.type
ORDER BY volatility DESC
----

.Neo4j Cypher: the traversal half
[source,cypher]
----
MATCH (a:Aircraft)-[:HAS_SYSTEM]->(s:System)
      -[:HAS_COMPONENT]->(c:Component)
      -[:HAS_EVENT]->(m:MaintenanceEvent)
WHERE a.tail_number = 'N95040A'
  AND m.severity = 'CRITICAL'
RETURN s.name, c.name, m.description, m.event_date
ORDER BY m.event_date DESC
----

The architecture treats Databricks and Neo4j as complementary systems connected through governed data pipelines. Counting stays in SQL. Following connections moves to the graph. Questions that span both route to specialized agents. See xref:lab4.adoc[Lab 4] for how the supervisor implements this routing.

== Lakehouse Architecture

=== The Medallion Pattern

Data moves through three progressive refinement layers:

* **Bronze** is the raw landing zone. Aircraft CSV files arrive from cloud storage (S3, ADLS Gen2, GCS) with no transformation. This is the system of record for what was ingested and when.
* **Silver** is the curation layer. Schema enforcement catches malformed records, column renaming normalizes conventions, and type casting ensures consistency. Silver tables are governed and ready for downstream consumers, including the Neo4j Spark Connector. Graph-bound data gets its last quality check here before projection.
* **Gold** is where graph intelligence feeds back into the lakehouse. Graph algorithm results (PageRank scores for component importance, community detection clusters for related maintenance events, degree centrality counts across the fleet topology) write back from Neo4j as columns in gold tables. These join with sensor telemetry and fleet metadata that never left the lakehouse to produce maintenance alerts, risk scores, and ML feature tables.

=== Delta Lake

Delta Lake provides ACID transactions, schema enforcement, and time travel on top of cloud object storage. For this architecture, it serves as the interchange format between Databricks and Neo4j. The Spark Connector reads from governed silver tables and writes graph algorithm results back to gold tables. Schema enforcement upstream means the graph receives clean, typed data without additional transformation at load time. Time travel enables rollback when a pipeline run corrupts data, avoiding full re-ingestion.

== The Spark Connector

The Neo4j Spark Connector is the Databricks-certified bridge for bulk data movement. It reads silver Delta tables and writes them as graph nodes and relationships. The transformation follows a consistent pattern: table rows become nodes with column values as properties, foreign keys become relationships, and shared attribute values (like an airport code appearing on multiple flights) become shared nodes that make hidden connections explicit. The connector also reads from Neo4j back into DataFrames for validation and bidirectional data flow.

See xref:lab2.adoc[Lab 2] for a hands-on walkthrough loading the aircraft digital twin data, including the specific transformation patterns for each entity type.

== Five Integration Patterns

Five patterns connect Databricks to Neo4j, each suited to a different workload.

=== ELT Data Pipeline

Governed Delta tables project into graph nodes and relationships via the Spark Connector. Silver tables feed the connector; the connector writes nodes and relationships into Neo4j using batched upserts. This is the primary path for bulk loading structured data.

=== Virtual Graph

The ELT data pipeline moves data into Neo4j. The virtual graph leaves it in place. A schema map defines a graph model over existing Delta tables, and a virtual graph database translates Cypher into SQL that pushes down to the lakehouse. Rows become nodes, foreign keys become relationships, and the query executes where the data already lives. No ETL job runs and no second copy of the data exists in Neo4j.

This inverts the Unity Catalog JDBC federation covered below. JDBC federation registers Neo4j as a SQL source so BI tools query the graph through SQL. The virtual graph registers the lakehouse as a graph source so Neo4j tooling queries Delta tables through Cypher. The full Neo4j Graph Intelligence Platform applies: Browser, Bloom, Copilot, and Graph Analytics operate against lakehouse data without a load step.

Four properties define the pattern:

* **Zero data movement**: data stays in the lakehouse under its existing storage and lifecycle.
* **Full pushdown**: Cypher compiles to SQL and computation runs in the lakehouse engine where the data lives.
* **Standard protocols**: Bolt and JDBC connect existing tools without changes.
* **Your governance**: Unity Catalog access policies on the underlying tables continue to apply.

The virtual graph suits exploration and analysis over data that already lives in the lakehouse, where the cost or latency of a full projection is hard to justify. When traversal performance at scale, write-back of graph algorithm results, or a persistent operational graph matters, the ELT data pipeline remains the right path.

=== Knowledge Graph Construction

Maintenance manuals, technical bulletins, and regulatory filings pass through a knowledge graph construction pipeline. Documents split into chunks, embedding models convert chunk text into vectors, and LLM-based entity extraction pulls structured nodes from the text. The `neo4j-graphrag-python` library's SimpleKGPipeline handles this pipeline using the Neo4j Python driver directly (chunking and entity extraction are not Spark operations). Extracted entities cross-link to the operational graph built by the data pipeline, bridging maintenance knowledge and fleet structure.

=== Data Analytics: Graph as Co-Processor

Graph algorithms run on in-memory projections in Neo4j and write results back to gold Delta tables through the Spark Connector:

* **PageRank** produces importance scores for components.
* **Community detection (Louvain)** identifies clusters of related maintenance events.
* **Degree centrality** counts connection density across the fleet topology.

These graph-derived metrics become columns in Delta tables, available for dashboards, ML features, and downstream analytics.

Unity Catalog JDBC connections add a governed SQL layer: BI tools like Power BI and Tableau query graph data through standard JDBC, and graph results join with lakehouse data without leaving the SQL environment.

=== GraphRAG and Agents: Multi-Platform Retrieval

Specialized agents query each platform independently. The Neo4j MCP agent discovers the graph schema through `get-schema` and executes read-only Cypher through `read-cypher`. The Genie agent translates natural language into SQL against the lakehouse. A supervisor agent routes questions based on their nature: aggregations and trends go to Genie, relationships and traversals go to the graph agent, and questions that span both platforms decompose into sub-tasks before the supervisor synthesizes a combined answer.

== Unity Catalog

For the GraphRAG architecture, Unity Catalog plays two specific roles:

* **Governance of graph-bound data**: the silver tables that feed the Spark Connector are governed assets with lineage tracking. When graph data quality issues surface, lineage traces back through the exact transformations and source records that produced the data.
* **Federated access to Neo4j**: Neo4j can be registered as a JDBC connection in Unity Catalog using the Neo4j JDBC driver. The driver translates SQL into Cypher automatically, so users and BI tools write SQL against what appears to be another catalog, and Neo4j executes the equivalent graph query. This enables federated queries that combine graph relationships and lakehouse metrics in a single SQL statement, governed under the same access controls as the rest of the lakehouse. Graph data can also be materialized as Delta tables for use in dashboards and Genie spaces. This is the reverse of the virtual graph pattern: JDBC federation translates SQL into Cypher to query the graph as SQL, while the virtual graph translates Cypher into SQL to query the lakehouse as a graph.

== Agent Systems

Databricks provides the agent development and serving stack through Mosaic AI, from foundation model access to multi-agent orchestration:

* **Foundation models** through the AI Gateway handle embedding generation for vector search and the reasoning layer that agents use to select tools and synthesize answers.
* **Genie** translates natural language into governed SQL against any data registered in Unity Catalog. Domain experts configure Genie Spaces with curated table sets, domain terminology, and example queries. In this architecture, Genie handles the sensor telemetry half of the multi-agent system: questions about averages, trends, and fleet-wide comparisons route to Genie, which generates SQL against the lakehouse. All generated queries are read-only.
* **Agent Bricks** is the control plane for deploying and orchestrating agents. The supervisor agent pattern is configurable through the workspace UI: register sub-agents (Genie spaces, MCP servers, Unity Catalog functions), provide capability descriptions, and the supervisor routes questions based on their nature. Agents package with MLflow, register in Unity Catalog, and deploy to Model Serving endpoints that scale automatically. Built-in tracing and feedback collection through the Review App support production monitoring.

See xref:lab4.adoc[Lab 4] for the hands-on implementation of Genie, the Neo4j MCP agent, and the supervisor.

== Agent Memory

Where Mosaic AI provides the agent development and serving stack, Neo4j contributes memory to the agents that run on it. Agent memory is a capability the supervisor and its sub-agents use, not a new connection between the platforms. It exposes three functions through the same agent-tool mechanism the Neo4j MCP agent already uses:

* **Short-term memory** holds working context within a session: recent turns, intermediate results, and the state the supervisor carries while it decomposes a multi-platform question and waits on its sub-agents.
* **Long-term memory** persists across sessions as a graph of facts, entities, and the relationships between them. An agent recalls prior findings and connects them to new ones by traversing that graph at retrieval time. Storing memory as a graph fits the shape of the data: memory is relational, and the connections between remembered facts are themselves what the agent reasons over.
* **Decision traces** capture the agent's routing choices, tool calls, and reasoning as a connected graph. Each decision links to the context that produced it, which supports auditability and lets later decisions reason over earlier ones.

Decision traces are distinct from the MLflow tracing in Agent Bricks. MLflow tracing is observability for monitoring runs and collecting feedback. Neo4j decision traces are a queryable graph the agent traverses at runtime and builds long-term memory from. The two are complementary: one watches the agent from the outside, the other gives the agent a memory it can reason over.

Agent memory plugs into Agent Bricks the same way the other sub-agents do. The supervisor reads and writes memory through agent tools, so no new connection type is introduced.
